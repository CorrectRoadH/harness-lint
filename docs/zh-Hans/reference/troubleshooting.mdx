本页覆盖使用 harness-lint 时最常见的错误和意外行为，并给出每个问题的分步修复方式。

<Tip>
  遇到任何问题时，第一步运行 `harness-lint doctor`。它会检查每个前置条件：项目根、配置文件、本地规则、已安装规则包、`grit` CLI 和 git 集成，并准确告诉你哪个检查失败以及原因。
</Tip>

<Accordion title="找不到 `grit`">

**症状：** `harness-lint check` 或 `harness-lint doctor` 打印类似错误：

```
[error] grit: grit is required for `check` but was not found: No such file or directory
```

**原因：** `grit` CLI 未安装，或不在你的 `PATH` 上。harness-lint 把所有 GritQL 模式执行委托给 `grit check`。

**修复：** 使用 Homebrew 安装 `grit`：

```sh
brew install getgrit/tap/grit
```

安装后，确认 harness-lint 能找到它：

```sh
harness-lint doctor
```

你应该在输出中看到 `[ok] grit: grit is installed: <version>`。如果你通过其他方式安装了 `grit`，请确保二进制位于 shell 的 `PATH` 中。

</Accordion>

<Accordion title="找不到 `harness.toml`">

**症状：** 任意 harness-lint 命令打印：

```
could not find project root from <path>; run `harness-lint init` first
```

或：

```
missing harness.toml; run `harness-lint init` in <path>
```

**原因：** harness-lint 会向上遍历目录树查找 `harness.toml` 或 `.git`。如果直到文件系统根目录都找不到，就会放弃。

**修复（新项目）：** 在仓库根目录运行 `harness-lint init`，创建 `harness.toml` 和默认 `rules/` 目录：

```sh
cd /path/to/your/repo
harness-lint init
```

**修复（目录不对）：** 如果你是在项目子目录中运行 harness-lint，使用 `--cwd` 指向仓库根目录：

```sh
harness-lint --cwd /path/to/your/repo check --changed
```

**修复（显式配置路径）：** 也可以直接指向配置文件：

```sh
harness-lint --config /path/to/harness.toml check --changed
```

</Accordion>

<Accordion title="规则没有可执行 GritQL">

**症状：** `harness-lint doctor` 或 `harness-lint rule verify` 打印：

```
validated N local rule(s), but only M include executable GritQL
```

或：

```
has a ```grit block but no executable GritQL
```

**原因：** 每个规则文件必须包含恰好一个带真实模式的 GritQL fenced code block。harness-lint 会拒绝空代码块、只包含注释的代码块，或只包含 `TODO` 行的代码块。

**修复：** 打开规则文件，在 `grit` 代码块中加入有效模式。代码块必须以 `language <name>` 指令开始，并跟随至少一个模式表达式：

````markdown
```grit
language python
`print($value)`
```
````

如果你还不能把约束表达成 GritQL 模式，不要创建 harness-lint 规则。把指导保留在 `AGENTS.md`、审查笔记或项目文档中。

添加模式后，验证它能工作：

```sh
harness-lint rule verify local.your-rule-id
```

</Accordion>

<Accordion title="规则包未安装">

**症状：** `harness-lint check` 或 `harness-lint doctor` 打印：

```
pack `typescript` is not installed; run `harness-lint update`
```

或：

```
pack `typescript` is missing from harness.lock; run `harness-lint update` first
```

**原因：** 规则包被列在 `harness.toml` 的 `[packs]` 下，但尚未获取到本地。fresh clone 后、队友添加规则包但没运行 `update`、或 `.harness/packs/` 被意外删除时会发生。

**修复（fresh clone 或缓存被删）：** 从已有 lock 文件恢复规则包：

```sh
harness-lint restore
```

**修复（lock 文件缺失或过期）：** 获取并锁定最新规则包版本：

```sh
harness-lint update
```

然后提交更新后的 `harness.lock`，避免队友和 CI 遇到同样错误。

</Accordion>

<Accordion title="误报：规则触发了不该触发的代码">

**症状：** `harness-lint check` 在规则不该应用的文件中报告违规，例如生成文件、测试 helper 或有意例外。

**方案 1：对已知且有文档说明的例外使用 `[[exceptions]]`：**

在 `harness.toml` 中添加条目，在特定路径上隐藏这条规则。始终填写 `reason`，让审查者和 AI 代理理解为什么这个抑制是有意的：

```toml
[[exceptions]]
rule   = "typescript.no-console-log"
paths  = ["src/generated/**"]
reason = "Generated SDK code emits debug output during local mocks."
```

其他规则仍会正常检查这些文件，只有指定规则被隐藏。

**方案 2：用 `$filename` 条件收窄 GritQL：**

如果你拥有这条规则，把文件范围编码进 GritQL 代码块。这比 `[[exceptions]]` 更精确，因为它自解释，也不需要配置变更：

```grit
language js
`console.log($value)` where {
  $filename <: r".*src/.*\.ts",
  !$filename <: r".*\.test\.ts"
}
```

**方案 3：用 `runs_on` + default-closed `[file_sets.*]` 把规则限定到/排除出某区域：**

如果误报全部来自一块成块的区域（典型是生成代码），把这块区域做成 default-closed file set，普通规则就再也看不到它。在 `harness.toml` 中：

```toml
[file_sets.generated]
paths = ["backend/gen/**/*.pb.go"]
default_rules = false
provides = ["generated"]
```

之后普通规则（无 `runs_on`）不再扫这块区域；只有在 frontmatter 里写 `runs_on: ["generated"]` 的专门规则才会进入。需要专门规则检查生成代码时用这个方案，而不是 `[ignore]`——`[ignore]` 不可破，无法被任何规则重新订阅。

**方案 4：对永远不应扫描的路径使用 `[ignore]`：**

如果没有任何规则应该扫描某个目录，例如 `dist/` 或 `coverage/`，把它加入 `[ignore]`：

```toml
[ignore]
paths = ["dist/**", "coverage/**"]
```

</Accordion>

<Accordion title="`harness.unknown-run-target`：规则 runs_on 的目标无人提供">

**症状：** `harness-lint doctor` 报告（锚定在 `harness.toml`）：

```
harness.unknown-run-target: rule `local.proto-no-id-getter` runs_on `generated`, but no file set provides it
```

**原因：** 某条规则的 `runs_on` 点名了一个 file set 名或概念，但没有任何 `[file_sets.*]` 提供它。常见于安装了出 `runs_on: ["<概念>"]` 的共享 pack，却没在项目里加匹配的 `[file_sets.*] provides = ["<概念>"]`。

**修复：** 在 `harness.toml` 里加一个带匹配 `provides` 的 file set，把概念连到项目的实际路径。pack 通常会在自己的 `INSTALL.md` 里声明所需概念：

```toml
[file_sets.generated]
paths = ["backend/gen/**/*.pb.go"]
default_rules = false
provides = ["generated"]
```

这是 `error`（默认）。可经 `[overrides]` 调级。

</Accordion>

<Accordion title="`harness.empty-file-set`：file set 没有 paths">

**症状：** `harness-lint doctor` 报告：

```
harness.empty-file-set: file set `generated` has no paths
```

**原因：** 某个 `[file_sets.<名字>]` 的 `paths` 缺失或为空。空集合不认领任何文件，没有意义。

**修复：** 给该集合至少加一个 repo 相对 glob：

```toml
[file_sets.generated]
paths = ["backend/gen/**/*.pb.go"]
default_rules = false
```

这是 `error`（默认）。

</Accordion>

<Accordion title="`harness.file-set-ignore-overlap`：file set 路径又被 [ignore] 覆盖">

**症状：** `harness-lint doctor` 报告：

```
harness.file-set-ignore-overlap: file set `generated` paths overlap `[ignore]`; those files are never reachable
```

**原因：** 某个 file set 的 `paths` 与 `[ignore].paths` 重叠。`[ignore]` 不可破——被它移除的文件无法被任何规则重新订阅，所以这块区域永远不可达，订阅它的规则什么也扫不到。

**修复：** 在两者中二选一。如果这块区域需要被专门规则检查，从 `[ignore]` 里移除对应路径，只保留 default-closed file set：

```toml
# 移除 [ignore] 中的 "backend/gen/**"，只留 file set
[file_sets.generated]
paths = ["backend/gen/**/*.pb.go"]
default_rules = false
provides = ["generated"]
```

如果确实没有任何规则该扫这块区域，删掉 file set，保留 `[ignore]`。这是 `error`（默认）。

</Accordion>

<Accordion title="`harness.stale-file-set-path`：file set 路径不存在">

**症状：** `harness-lint doctor` 报告：

```
harness.stale-file-set-path: file set `generated` path `backend/gen/**/*.pb.go` matches nothing
```

**原因：** 某个 file set 的 `paths` 条目的字面量前缀在磁盘上已不存在——通常是目录被重命名或删除后没更新配置。陈旧条目悄悄不再匹配，订阅该区域的规则不再扫到预期文件。

**修复：** 更新该路径指向当前位置，或删掉过时条目：

```toml
[file_sets.generated]
paths = ["backend/proto/gen/**/*.pb.go"]
default_rules = false
```

这是 `warn`（默认），可经 `[overrides]` 升级为 `error`。首组件本身就是 glob 的模式（如 `**/*.pb.go`）不检查，因为没有可锚定的字面量前缀。

</Accordion>

<Accordion title="`harness.runs-on-filename-disjoint`：runs_on 与 $filename 不相交">

**症状：** `harness-lint check` 或 `harness-lint doctor` 报告（doctor 里归在 `runs-on-scope` 下）：

```
harness.runs-on-filename-disjoint: rule `local.proto-no-id-getter` runs_on file set never overlaps its `$filename` regex; the rule matches nothing
```

一条规则同时用了 `runs_on`（file-set 作用域）和正向 GritQL `$filename` 正则，却什么都匹配不到——静默扫 0 个文件。

**原因：** `runs_on` 在扫描阶段选文件；GritQL 的 `$filename` 谓词再在 grit 内部过滤被选中的文件。如果 `runs_on` 区域和 `$filename` 作用域**不相交**，被选中的文件全被 `$filename` 拒掉 → 0 命中。

**修复：** 删掉 `$filename` 里多余的区域判断——`runs_on` 已经把范围限定到 file set，`$filename` 只该保留 `runs_on` 表达不了的东西，比如区域内的子收窄或 `!` 排除：

```grit
language go
`$x.GetId()` where {
  $runs_on,                          # runs_on 已选好 file set
  !$filename <: r".*_mock\.pb\.go"   # 仅保留 runs_on 无法表达的排除
}
```

或者把 `$filename` 正则和 file set 的 `paths` 对齐，让两者相交。

**保守触发：** 仅在通过字面路径前缀可证两者不相交时才触发。`runs_on` 含 `default`、无正向 `$filename`、或仅有 `!` 排除时都不触发。这是 `warn`（默认），可经 `[overrides]` 按 id 调级。

</Accordion>

<Accordion title="`check` 拒绝扫描：文件太多">

**症状：** 不带范围 flag 运行 `harness-lint check` 时打印：

```
refusing implicit full scan of N files; use `harness-lint check --changed` or run `harness-lint check --all` to force it
```

**原因：** 当你不带 `--changed`、`--staged` 或 `--all` 运行 `harness-lint check` 时，harness-lint 会执行隐式完整扫描。为了避免意外扫描大型项目，当文件数超过 1,000 时它会拒绝。

**修复（多数工作流推荐）：** 只扫描相对基准分支发生变化的文件。这是 pre-push 检查最快的选择：

```sh
harness-lint check --changed
```

**修复（CI 完整扫描）：** 如果你确实要扫描每个文件，例如夜间 CI job，显式传 `--all`：

```sh
harness-lint check --all
```

**修复（pre-commit）：** 只扫描 staged 文件：

```sh
harness-lint check --staged
```

</Accordion>

<Accordion title="`rule verify` 失败：Bad 示例没有触发">

**症状：** `harness-lint rule verify` 打印：

```
Bad example 1 for rule `local.your-rule` did not trigger; adjust the GritQL, Bad example, or `$filename` scope
```

**原因：** GritQL 模式没有对 `## Bad` 示例代码产生匹配。通常有三种原因：

1. GritQL 模式太具体，没有匹配 Bad 示例的形状。
2. 模式中的 `$filename` 条件排除了验证时使用的合成文件名（`src/bad-example.<ext>`）。
3. Bad 示例代码对该规则语言而言语法无效。

**修复：检查模式：**

打开规则文件，对比 GritQL 模式和 Bad 示例。运行 `harness-lint rule explain <rule-id>` 并并排查看两者。简化模式，或扩展 Bad 示例让它匹配。

**修复：检查 `$filename` 范围：**

`harness-lint rule verify` 会把 Bad 示例写入临时文件，路径为 `src/bad-example.<ext>`。如果模式使用 `$filename` 条件，确认它会匹配类似 `src/bad-example.ts` 的路径：

```grit
language js
`console.log($value)` where {
  $filename <: r".*src/.*\.ts"   ✅ matches src/bad-example.ts
}
```

像 `$filename <: r".*apps/.*\.ts"` 这样的条件会排除验证文件。开发规则时，可以调整条件或临时放宽它。

**修复：检查 Bad 示例语言：**

确认 `## Bad` 中的 fenced code block 使用正确语言标识符，并且代码对该语言而言语法有效。

</Accordion>

# 方案 4：domains —— 按 owner 拆分扫描范围,用"命名域 + 概念订阅"取代全局 ignore 减法

相关文档：[GOALS](./GOALS.md) · [PLAN-1](./PLAN-1.md) · [PLAN-2](./PLAN-2.md) · [PLAN-3](./PLAN-3.md) · [DECISION](./DECISION.md)

Status: **核心洞察被采用**（owner 拆分 + 白名单判据由 [PLAN-5](./PLAN-5.md) 以用户语言落地；[DECISION](./DECISION.md) 采用 PLAN-5）

这篇从 0 重新设计"一条规则扫描哪些文件"。它不是在 PLAN-1/2/3 上加字段,而是换掉会导致级联的底层原语,并按**谁拥有哪份知识**来切分配置。PLAN-1/2/3 作为"被否决的减法/打洞路线"保留在文末。

## 从用户场景开始

规则有两种来源:**installed(安装的共享 pack 规则)** 和 **local(项目自写规则)**。它们的扫描范围由谁决定,差别很大:

| 场景 | 规则来源 | "扫哪里"的知识在谁手里 |
|---|---|---|
| A. 自写 proto 规则查 `GetId()` | local | 项目自己:既知道规则意图,也知道 `backend/gen` 在哪 |
| B. 装了 `go-protobuf` pack,里面的 proto 规则 | installed | **裂开**:pack 作者知道"只对生成的 `.pb.go` 成立",但不知道你把 gen 放哪;项目知道 gen 在 `backend/gen`、且不该被普通规则扫 |
| C. 装了 `no-console-log` pack | installed | 普通源码,跑默认范围即可,项目零配置 |
| D. 装的通用规则,项目想让它额外扫某段 / 别扫某段 | installed | 纯项目布局知识,只有项目知道 |

场景 B 是核心教训:**一条 installed 规则的扫描范围是两份独立知识,属于两个人。**

- **"这规则只对什么文件类成立"** —— 规则的**内在身份**(`*.pb.go`、带 `DO NOT EDIT` 头)。**pack 作者拥有**,且必须用**可移植的词**(文件名 pattern / 内容)表达,**绝不能是项目路径**——他不知道项目布局。
- **"我项目里这些路径,多数规则别看"** —— **项目布局**。**项目拥有**,用路径表达。

任何把这两份知识压进同一个字段的设计(PLAN-2 的 `scan_only` 同时表达 scope 和路径;PLAN-4 早期稿让 pack 规则去订阅项目命名的域)都会逼一方迁就另一方,产生摩擦或命名冲突。本方案按 owner 把它们拆开。

## 病根:错形状的原语会强制级联

PLAN-1/2/3 都建立在"`[ignore]` 是全局的、与规则无关的减法"之上。一旦"全局减法"遇上"个别规则要例外",级联是数学必然:

```
[ignore] 减法 → scan_only 打洞(反减) → [[scan_ignored]] 授权(反反减) → local/pack 优先级阶梯(裁决)
```

这是 CSS specificity、也是 ESLint 旧 `.eslintignore` + `overrides` + `!negation` 的同一种病。补丁加不完,因为问题被塞进了减法原语。

## 核心判据:什么时候用黑名单,什么时候用白名单

> **默认该被扫(default-open)→ 黑名单。默认不该被扫(default-closed)→ 白名单。**

这条判据决定整个架构:

- **普通源码 = default-open ⇒ 黑名单。** 你希望**新加文件自动被 lint**,漏扫比误扫危险。所以:扫一切,除了项目显式排除的。排除是项目侧、布局知识。
- **生成 / 排除区 = default-closed ⇒ 白名单。** 你**不希望**任何规则不小心扫进 gen,更不希望一条 installed 规则**静默穿透**你的排除。所以:只有显式声明"我就是冲这类文件来的"规则才能进。"我冲这类来"是规则侧、可移植;"这类在哪"是项目侧。

PLAN-2 的错本质上是:把"穿透排除区"这个 default-closed 的特权,做成了规则单方面就能开的黑名单逃逸。

## 模型

### 第 0 层:结构排除(唯一的全局减法,不可破)

```text
visible source = current run set
              − .git / .harness / 规则目录 / node_modules / target / 构建产物
              − .gitignore 命中
```

任何字段都不能突破这一层。它真正普世,名副其实地全局。模型里**只有这一处**全局减法。

### 项目侧:命名域(default-closed 区域的"在哪")

项目在 `harness.toml` 里命名"多数规则不该看、但某些规则需要"的文件类:

```toml
[domains.generated]
paths = ["backend/gen/**", "**/*.pb.go"]

[domains.migrations]
paths = ["**/migrations/*.sql"]
```

域是项目布局知识,用路径表达,owner 是项目。

### 总划分(partition)

```text
对 visible source 里每个文件 f：
  f ∈ 命名域 D     当 f 命中 D.paths（可同时属于多个域）
  f ∈ default     当 f 不命中任何命名域
```

`default` 不是配置项,是"没被任何命名域认领的剩余源码"。

### 规则侧:portable 正向 scope(default-closed 区域的"我冲谁来")

规则用**可移植**的方式声明自己的内在身份——绝不含项目路径:

```markdown
---
id: go-protobuf.require-id-getter
language: go
applies_to: generated          # 概念标签:我属于"生成代码"这类
$filename: r".*\.pb\.go"        # 可移植内在身份(文件名 pattern / 内容),不含项目路径
---
```

- `applies_to` 是**概念标签**,不是路径。owner 是规则/pack 作者,跟着规则走。
- 无 `applies_to` ⇒ 规则只跑 `default`(场景 C:普通 installed 规则零配置)。
- `applies_to: [default, generated]` ⇒ 默认源码 + 生成代码都跑(PLAN-1 的 additive 场景,无需新机制)。

### 绑定:按概念交集,自动完成

```text
规则 R 能看到的文件 =
    ⋃( R.applies_to 里每个标签解析出的项目域 )     // 默认标签 = default 域
  ∩ R.language 匹配
  ∩ R.$filename 谓词
```

- pack 作者只说"我属于 `generated`"(portable);项目只说"`generated` = `backend/gen`"(布局)。二者在 `generated` **这个概念上交汇**,绑定自动发生。pack 作者没碰项目路径,项目没碰规则。
- 绑定**按概念/标签**,不是逐规则授权:定义一次 `generated`,所有"冲生成代码来的" pack 规则**自动归位**——这正是避开级联的关键。
- 规则 `applies_to` 引用项目**未定义**的域 ⇒ **报错**(响亮),项目决定补一行还是弃用该规则。
- pack 规则**不可能**看到项目没划进任何域的文件 ⇒ 主权结构性保住,**不需要 PLAN-3 的 local/pack 授权阶梯**;local 与 pack 在扫描语义里不再有区别。

并集满足交换律/结合律 ⇒ 没有 specificity、没有顺序依赖、没有"谁盖过谁"。

### 自动识别:把命名摩擦降到零

`generated` 这类概念是**自识别**的:`.pb.go` 带 `// Code generated by ... DO NOT EDIT.` 头,且通常已在 `.gitignore` 或 `[ignore]`。harness 可内建对 `DO NOT EDIT` 头的 `generated` 自动识别,常见情况下项目**连 `[domains.generated]` 都不用写**。只有项目私有的特殊类(如 `legacy-module`)才需手写域。

## 四个场景端到端

```toml
# harness.toml —— 多数情况下甚至不需要,DO NOT EDIT 头自动识别 generated
[domains.generated]
paths = ["backend/gen/**/*.pb.go"]
```

- **A. local proto 规则**:`applies_to: generated` + `$filename: *.pb.go`。和写 local 规则一样自然。
- **B. installed proto 规则**:pack 出厂带 `applies_to: generated` + `$filename`。项目装上即用;它只看项目的 `generated` 域,不碰别处。pack 作者从未硬编码项目路径。
- **C. installed 通用规则**:无 `applies_to`,跑 `default`,看不到 `generated`,**零配置**。
- **D. 项目覆盖 installed 规则范围**:别扫某段 → `[[exceptions]]`(报告阶段)或把那段划进它不订阅的域;额外扫某段 → 把那段并进它 `applies_to` 的域。**都不改 pack。**

## 与 `[ignore]` 的兼容:它只是"不可订阅的域"

`[ignore]` 不作为第二机制保留,而是 desugar 成域模型的特例:**一个 `subscribable = false` 的域**——任何规则都不能 `applies_to` 它。这保留了"真正的垃圾应永不可扫"这一安全属性,同时让常见的"直接忽略"保持简短。

```text
[ignore].paths = [...]   ≡   一个 subscribable = false 的命名域
```

一个模型(域),一个布尔(可否订阅)。现有 `[ignore]` 配置零破坏。

> 经验法则:文件**永远不该被任何规则扫** → `[ignore]`(或本就该进 `.gitignore` / 结构排除);文件**多数规则跳过、个别规则需要** → `[domains.*]`。

## 完整优先级阶梯(单调,可讲)

```text
1. 结构排除 + .gitignore                ← 最高,任何字段不可破
2. current run set (--changed/--staged/--all)
3. 域划分:f → default 或命名域;[ignore] = 不可订阅域
4. ∩ ⋃(规则 applies_to 解析出的域)        ← 并集,无 specificity
5. ∩ language ∩ $filename
```

## 三阶段模型(不变,更干净)

| 阶段 | 负责 | 配置 |
|---|---|---|
| 启用 | 规则启用/禁用/severity | `[disabled]`、`[overrides]` |
| **扫描** | 规则能看到哪些文件 | **`[domains.*]`(项目侧路径) + 规则 `applies_to`(portable 概念) + `language` + `$filename`**,结构排除在最前 |
| 报告 | 诊断是否显示 | `[[exceptions]]` |

domains 纯属扫描阶段;`[[exceptions]]` 仍只隐藏诊断、不改扫描范围。

## 健康检查(doctor / rule validation）

- 域 `paths` 的 stale 前缀检查:复用现有 `harness.stale-ignore-path` 同款逻辑,锚到 `harness.toml`。
- 规则 `applies_to` 引用**未定义**的域 ⇒ error,锚到规则文件(抓拼写错误,也抓 pack 规则期待项目未提供的域)。
- 规则 `applies_to` 引用 `subscribable = false` 的域(如 `[ignore]` 派生域)⇒ error,锚到规则文件。
- 命名域定义了但**无任何规则订阅** ⇒ info/warn(可能死配置,或就是当 `[ignore]` 用——可静音)。
- `applies_to` 为空列表 ⇒ error。

## 老实说的代价 / red-team

1. **定义域会隐式改变 `default`。** 定义 `[domains.generated]` 后,所有 `default` 规则立刻看不到那些文件。这是 GOALS 想要的(gen 默认隐身),且它是一次**总划分**(并集),不是优先级链,不会级联。不想要的规则显式 `applies_to: [default, generated]` 加回来。这是唯一需要文档重点讲清的"魔法"。
2. **需要概念词表。** `applies_to` 的标签是 pack 与项目之间的共享词汇(`generated`、`migrations`…)。内建几个自识别概念(尤其 `generated`)能覆盖绝大多数情况;项目私有概念自定义。需要一份推荐标签清单,避免每个 pack 各造新词。
3. **`.gitignore` 仍不可破。** 若生成代码是 `.gitignore` 而非提交进仓,domains 和 PLAN-1/2/3 一样够不到——结构层故意绝对。这是 GOALS 该写明的**前提**(gen 代码须 committed,放命名域而非 `.gitignore`)。
4. **概念数。** 相对今天(`[ignore]` + 逐规则 scope)是平移;相对 1→2→3 的轨迹是大幅净简化,并顺带消掉 PLAN-1 的存在理由、PLAN-2 的 bypass 语义、PLAN-3 的授权阶梯。卖点是后者。

## 被否决的路线(PLAN-1 / 2 / 3)

三者共享同一个会级联的前提:把 `[ignore]` 当全局减法,再逐规则打洞。

- **[PLAN-1](./PLAN-1.md)**(`[[scan_ignored]]`,toml,additive):把一条规则的范围拆到两个文件;须解释"为何盖过 ignore";对 gen-only 表达的是"默认+gen"而非"只 gen"。
- **[PLAN-2](./PLAN-2.md)**(`scan_only`,rule,exclusive):一个字段同时干 scope 和 ignore-bypass;**任何规则(含 pack)声明它即可静默穿透项目 `[ignore]`,项目无否决权**——主权漏洞,也正是 default-closed 特权被做成黑名单逃逸的错。
- **[PLAN-3](./PLAN-3.md)**(按 local/pack 授权):修了 PLAN-2 的漏洞,但靠再加一层"谁有权突破"的优先级阶梯——补丁修补丁,CSS 化。

domains 不修补这条链,而是移除它的前提(全局减法),并按 owner 把 portable scope(规则侧)与布局(项目侧)拆开,因此三者要解决的问题在新模型里**不再产生**。

## 落地建议

- 把 `DECISION.md` 改为采用 domains;PLAN-1/2/3 标 Superseded,DECISION 留一节"被否决的级联路线"指向它们。
- `GOALS.md` 增补"gen 代码须 committed(放命名域),`.gitignore` 的生成代码不在本设计范围"的前提。
- 定义内建概念标签(至少 `generated`,含 `DO NOT EDIT` 自动识别)与推荐词表。
- README 配置组合改用三阶段模型,扫描阶段以 domains + `applies_to` 为中心讲;`[ignore]` 文档说明它是"不可订阅的域"。
- doctor / rule validation 落地上述健康检查。

## 取舍了哪些用户旅程

参见 [GOALS](./GOALS.md#用户旅程) J1–J7。本方案为 J3 / J4(两个现实约束)而生,代价压在 J2。

| 旅程 | 体验 | 原因 |
|---|---|---|
| J1 onboarding | 顺 | 普通规则无 `applies_to` 即跑 default;gen 可由 `DO NOT EDIT` 头自动识别,常见情况连 `[domains.generated]` 都不写 |
| J2 local gen-only | **中(比 PLAN-2 贵)** | 要定义域 + 规则订阅(两处);还要理解“定义 default-closed 域会从所有 default 规则手里静默拿走这些文件” |
| J3 pack 冲 gen | **最佳** | pack 写 `applies_to: generated`(可移植概念),项目写 `[domains.generated] = 路径`,**按概念自动绑定**。定义一次 `generated`,所有冲 gen 来的 pack 规则自动归位,无逐条授权 |
| J4 monorepo 散落 | **最佳** | 一个 `[domains.generated]` 列三处散落路径,所有 gen 规则(local + pack)一并绑定 |
| J5 改写 pack 范围 | 顺 | 加路径 = 改域;减报告 = `[[exceptions]]`。都不碰 pack |
| J6 目录搬家 | **最佳** | gen 搬家 = 改一处域定义;stale 检查锚到 `harness.toml` |
| J7 主权 | **结构性安全** | pack 看不到未被划进任何可订阅域的文件;`[ignore]` = 不可订阅域。不需要 PLAN-3 的授权阶梯 |

**买到的旅程**:J3、J4、J6、J7——跨来源可移植、散落同概念定义一次、搬家改一处、主权天然成立。这四条正是 PLAN-1/2/3 集体最弱的地方。
**牺牲的旅程**:J2——动机场景从 PLAN-2 的一个字段涨到两处编辑,外加“定义域静默减扫 default”这个唯一的魔法需要文档重点讲。
**残留风险**:共享概念词表是 pack 与项目之间**不可强制**的约定,且无别名机制——pack 说 `generated`、项目命名 `gen` 就要么报错要么用不了。这是整条 PLAN-4/5 路线的阿喀琉斯之踵。
